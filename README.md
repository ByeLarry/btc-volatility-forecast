# Прогнозирование дневной реализованной волатильности BTC

Данный репозиторий содержит реализацию исследования по прогнозированию дневной реализованной волатильности BTC на один день вперед. В работе используются 5-минутные рыночные данные, суточные сетевые показатели блокчейна, оконное формирование обучающих примеров и сравнение моделей MLP и LSTM при одинаковой схеме временной проверки.

## Общая логика

Сначала загружаются исходные 5-минутные ценовые данные BTC (OHLCV) и, при необходимости, суточные сетевые показатели блокчейна. Рыночные данные приводятся к UTC-времени, проверяются на наличие некорректных значений и дубликатов, после чего по 5-минутным логарифмическим доходностям рассчитываются дневная реализованная дисперсия и дневная реализованная волатильность. Неполные UTC-сутки исключаются из набора данных.

Далее рыночные и сетевые данные согласуются по UTC-дате. Внутренние пропуски суточного календаря линейно интерполируются, а граничные пропуски, которые нельзя восстановить, исключаются. На основе полученного суточного набора формируются два варианта признаков: базовый, включающий только прошлые значения дневной реализованной волатильности, и расширенный, дополненный метриками блокчейна. Целевая переменная и признаки преобразуются в логарифмическую шкалу, после чего признаки масштабируются отдельно на каждом обучающем интервале, чтобы исключить утечку информации из будущих периодов.

Проверка качества выполняется по схеме скользящего окна. На каждом шаге выделяется обучающий интервал, следующий за ним тестовый интервал, формируются перекрывающиеся временные последовательности заданной длины, затем обучаются модели MLP и LSTM. После получения прогноза выполняется обратное преобразование из логарифмической шкалы, рассчитываются выбранные в `experiment.metrics` показатели из набора MAE, MSE, RMSE, MAPE, QLIKE, R2 и DA, а результаты сохраняются в виде таблиц прогнозов, сводных метрик и графиков. QLIKE рассчитывается по фактической и прогнозной реализованной дисперсии, полученной возведением волатильности в квадрат.

## Основные этапы пайплайна

1. Сбор и загрузка исходных данных.
2. Подготовка 5-минутных рыночных данных и приведение времени к UTC.
3. Расчет дневной реализованной волатильности.
4. Согласование рыночных и сетевых данных по UTC-суткам.
5. Формирование базового и расширенного наборов признаков.
6. Логарифмическое преобразование и масштабирование признаков.
7. Формирование обучающих примеров на основе скользящих окон.
8. Обучение моделей MLP и LSTM.
9. Прогнозирование дневной реализованной волатильности на один день вперед.
10. Расчет метрик качества и сохранение результатов эксперимента.

## Входные данные

Для запуска эксперимента используются CSV-файлы, пути к которым задаются в `config.json`:

- 5-минутные рыночные данные BTC с колонками `timestamp`, `open`, `high`, `low`, `close`, `volume`;
- суточные сетевые показатели с колонками `date`, `unique_addresses`, `transfer_volume_btc`, `avg_fee_usd`.

## Результаты

После выполнения эксперимента сохраняются:

- прогнозы моделей по тестовым интервалам;
- метрики качества по каждому шагу скользящей проверки;
- сводная таблица средних значений, стандартных отклонений и медиан метрик;
- графики фактической и прогнозной реализованной волатильности;
- графики сравнения средних и медианных метрик моделей и наборов признаков;
- диаграммы размаха выбранных метрик по тестовым интервалам для каждой модели и набора признаков;
- при включенном итоговом обучении — финальные прогнозы, метрики и график на последнем тестовом окне;
- при включенном сохранении итоговой модели — модель Keras, преобразователь масштаба и файл метаданных.

Повторный запуск с тем же `paths.results_dir` перезаписывает CSV-файлы и графики с совпадающими именами. Для отдельного эксперимента следует задавать новый каталог результатов.

## Установка

В Windows окружение и зависимости можно установить командой:

```bat
install.bat
```

Скрипт создает виртуальное окружение `venv` рядом с проектом и устанавливает версии пакетов из `requirements.txt`.

## Запуск

Перед запуском необходимо подготовить корневой `config.json`, указав пути к данным, параметры окон, список моделей, наборы признаков и параметры обучения. Команду следует выполнять из каталога проекта: `main.py` по умолчанию читает `config.json` из текущего рабочего каталога.

Относительные пути внутри конфигурации разрешаются относительно каталога, в котором находится используемый файл конфигурации. Поэтому конфигурацию из каталога `results` нельзя использовать как корневую без проверки и корректировки относительных путей.

```bash
python main.py
```

## Описание параметров `config.json`

Все основные параметры эксперимента задаются в файле `config.json`. Конфигурация определяет пути к данным, параметры скользящей временной проверки, состав признаков, список моделей, настройки обучения и правила сохранения результатов.

Для поиска и сравнения конфигураций рекомендуется задавать `train_final_model: false`. После фиксации гиперпараметров параметр можно переключить в `true`, чтобы выбрать лучшую пару «модель + набор признаков» по `selection_metric` и выполнить итоговое обучение. Значение `results_dir` для каждого запуска должно быть уникальным.

### `paths`

Раздел задает пути к исходным данным и результатам эксперимента.

| Параметр | Описание |
|---|---|
| `market_5m_csv` | Путь к CSV-файлу с исходными 5-минутными рыночными данными BTC. Файл должен содержать колонки `timestamp`, `open`, `high`, `low`, `close`, `volume`. |
| `network_daily_csv` | Путь к CSV-файлу с суточными сетевыми показателями блокчейна. Ключ обязателен в конфигурации, но файл загружается только при наличии набора `extended` в `feature_sets`. |
| `daily_dataset_csv` | Путь, по которому сохраняется сформированный итоговый суточный набор данных. Файл перезаписывается при каждом запуске. |
| `results_dir` | Каталог, в который сохраняются прогнозы, метрики, графики и, при необходимости, итоговые модельные артефакты. При повторном использовании каталога одноименные файлы перезаписываются. |

### `experiment`

Раздел задает основную логику эксперимента и схему временной проверки.

| Параметр | Описание |
|---|---|
| `input_window` | Длина входного окна в сутках. При значении `30` модель использует предыдущие 30 дневных наблюдений для построения прогноза. |
| `forecast_horizon` | Горизонт прогноза в сутках. Текущая реализация поддерживает только значение `1`. |
| `train_window` | Размер обучающего интервала в сутках на каждом шаге скользящей проверки. Значение должно быть больше `input_window`. |
| `test_window` | Размер тестового интервала в сутках на каждом шаге проверки. При значении `30` модель проверяется на следующих 30 днях после обучающего окна. |
| `step_size` | Положительный шаг сдвига скользящего окна. При значении `30` следующее обучающее и тестовое окно сдвигаются на 30 дней. Если шаг больше `test_window`, между тестовыми интервалами остаются непроверяемые даты; если меньше — тестовые интервалы перекрываются. |
| `feature_sets` | Список наборов признаков для сравнения. `base` использует только прошлые значения реализованной волатильности, `extended` дополнительно использует сетевые показатели блокчейна. |
| `models` | Список моделей, участвующих в эксперименте. В текущей версии используются `mlp` и `lstm`. |
| `scaler` | Способ масштабирования признаков. Значение `standard` соответствует стандартизации, значение `minmax` — масштабированию в заданный диапазон. Масштабирование на каждом шаге настраивается только по обучающему интервалу. |
| `epsilon` | Малая положительная добавка, используемая при логарифмировании, обратном преобразовании и расчете метрик, чтобы избежать ошибок при нулевых или слишком малых значениях волатильности. Значение должно быть больше нуля. |
| `random_seed` | Базовое значение генератора случайных чисел. На шаге скользящей проверки используется сумма `random_seed + fold`; при итоговом обучении используется исходное значение. |
| `metrics` | Непустой массив рассчитываемых метрик. Допустимы `mae`, `mse`, `rmse`, `mape`, `qlike`, `r2`, `da` без учета регистра. Метрики вне массива не рассчитываются, не сохраняются в CSV и не выводятся на сравнительные графики и диаграммы размаха. Повторы и неизвестные значения запрещены. |
| `selection_metric` | Метрика выбора итоговой конфигурации. Она обязательно должна входить в `metrics`. Краткое значение, например `rmse`, означает среднее по фолдам; также поддерживаются префиксы `mean_`, `median_` и `std_`, например `median_RMSE`. Для средних и медианных `R2` и `DA` выбирается максимум, для остальных вариантов — минимум. Если параметр отсутствует, используется первая метрика из `metrics`. |
| `train_final_model` | Если `false`, выполняется только скользящая проверка. Если `true`, после нее выбирается лучшая конфигурация, модель обучается на всем доступном диапазоне до последнего окна длиной `test_window`, а последнее окно используется для финальной оценки. Если параметр отсутствует, используется `true`. |

### `training`

Раздел задает параметры обучения нейросетевых моделей. Функция потерь настраивается независимо от `selection_metric`: первая управляет обновлением весов внутри одного запуска, а вторая выбирает лучшую конфигурацию по результатам скользящей проверки.

| Параметр | Описание |
|---|---|
| `loss` | Функция потерь для MLP и LSTM. Допустимы `mse`, `mae`, `huber`, `qlike`. Значения `mse`, `mae` и `huber` применяются к прогнозируемому логарифму волатильности; `qlike` оптимизирует QLIKE по регуляризованной дисперсии, восстановленной из логарифмической шкалы. Если параметр отсутствует, используется `mse`. |
| `learning_rate` | Скорость обучения оптимизатора Adam. Чем меньше значение, тем осторожнее обновляются веса модели. |
| `batch_size` | Размер мини-пакета наблюдений при обучении. |
| `epochs` | Максимальное количество эпох обучения. Если включена ранняя остановка, обучение может завершиться раньше. |
| `validation_split` | Доля конца обучающей выборки, используемая для контроля качества при скользящей проверке. Перемешивание отключено. Для итоговой модели значение принудительно равно `0`, чтобы использовать весь доступный обучающий диапазон. |
| `early_stopping` | Включает или отключает раннюю остановку обучения. Если значение `true`, обучение прекращается при отсутствии улучшения ошибки. |
| `early_stopping_patience` | Количество эпох без улучшения, после которого обучение останавливается. |
| `verbose` | Уровень вывода информации во время обучения. Значение `0` отключает подробный вывод. |

### `mlp`

Раздел задает архитектуру модели MLP. Для этой модели входное временное окно предварительно преобразуется в плоский вектор признаков.

| Параметр | Описание |
|---|---|
| `hidden_units` | Количество нейронов в скрытых полносвязных слоях. Например, `[32, 16]` означает два скрытых слоя: первый на 32 нейрона, второй на 16 нейронов. |
| `dropout` | Доля случайно отключаемых нейронов после скрытых слоев. Используется для снижения переобучения. |
| `activation` | Функция активации скрытых слоев. |

### `lstm`

Раздел задает архитектуру модели LSTM, которая обрабатывает входные данные как последовательность дневных наблюдений.

| Параметр | Описание |
|---|---|
| `units` | Количество LSTM-слоев и число нейронов в каждом из них. Например, `[32]` означает один LSTM-слой на 32 нейрона. |
| `dropout` | Доля отключаемых входных связей внутри LSTM-слоя. Используется для регуляризации модели. |
| `recurrent_dropout` | Доля отключаемых рекуррентных связей внутри LSTM-слоя. |
| `dense_units` | Количество нейронов в полносвязных слоях после LSTM-блока. Например, `[8]` означает один дополнительный полносвязный слой на 8 нейронов. |
| `activation` | Функция активации полносвязных слоев после LSTM-блока. |

### `saving`

Раздел определяет, какие результаты сохраняются после выполнения эксперимента.

| Параметр | Описание |
|---|---|
| `save_models` | Если `true` и включен `train_final_model`, сохраняются выбранная итоговая модель, преобразователь масштаба и метаданные. Модели отдельных фолдов скользящей проверки не сохраняются. |
| `save_predictions` | Если `true`, сохраняется общая таблица прогнозов скользящей проверки. Финальные прогнозы при `train_final_model: true` сохраняются отдельно независимо от этого параметра. |
| `save_metrics` | Если `true`, сохраняются метрики по каждому фолду и сводная таблица со средними значениями, стандартными отклонениями и медианами. Финальные метрики при `train_final_model: true` сохраняются отдельно независимо от этого параметра. |
| `save_plots` | Если `true`, сохраняются графики скользящей проверки: фактические и прогнозные значения, сравнения средних и медианных метрик, а также отдельная диаграмма размаха для каждой метрики из `experiment.metrics`. На диаграммах отмечаются медиана, среднее значение и выбросы. Для положительных метрик ошибок используется логарифмическая шкала; если встречается нулевое значение, сохраняется линейная шкала. Метрики `R2` и `DA` отображаются в линейной шкале. Финальный график при `train_final_model: true` сохраняется отдельно. |
